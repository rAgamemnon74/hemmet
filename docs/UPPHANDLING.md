# Upphandlingar — designanalys

## Vad är en upphandling?

En upphandling är processen från **identifierat behov** till **tecknat avtal**. Det gäller inte bara hantverkare — allt föreningen köper externt är en upphandling:

- Reparera ett tak (fysisk tjänst, engång)
- Byta städfirma (driftstjänst, löpande)
- Välja bredband till 100 lägenheter (infrastruktur, 3–5 år)
- Byta bokföringssystem (IT-tjänst, SaaS)
- Omförhandla fastighetslån (finansiell tjänst)
- Upphandla styrelseförsäkring (risk, årlig)

Samma process — identifiera behov, utvärdera alternativ, besluta, teckna avtal — men med helt olika aktörer, kriterier och tempo.

## Upphandlingskategorier

### 1. Fysisk förvaltning — traditionella upphandlingar

Entreprenörer som arbetar på fastigheten.

| Underkategori | Exempel | Typisk process |
|---------------|---------|----------------|
| Akut reparation | VVS-läcka, låsbyte, hisstopp | Direkt → ring jour → faktura i efterhand |
| Planerat underhåll | Fasadmålning, fönsterbyte, takbyte | Spec → 3 offerter → styrelsebeslut → projekt |
| Stambyte/storskaligt | Stamrenovering, dränering | Konsult → spec → 3–5 offerter → stämmobeslut |
| Löpande drift | Städ, trädgård, snöröjning | Utvärdera befintligt → 2–3 offerter → nytt avtal |

**Karaktäristik:**
- Betalning: faktura, bankgiro
- Utvärdering: pris, tidsplan, garanti, referensobjekt
- Kontakt: telefon, platsbesök, fysisk inspektion
- Aktörer: förvaltare driver, leverantör besöker fastigheten

### 2. Tjänsteförvaltning — professionella tjänster

Specialister som föreningen anlitar för expertis.

| Underkategori | Exempel | Typisk process |
|---------------|---------|----------------|
| Ekonomisk förvaltning | Nabo, HSB, Riksbyggen, SBC | Utvärdera 2–3 → styrelsebeslut → 1–3 år |
| Teknisk förvaltning | Driftbolag, jourentreprenör | Utvärdera → avtal 1–3 år |
| Revision | Revisionsbyrå, lekmannarevisor | Stämman väljer → styrelsen tecknar avtal |
| Juridik | Fastighetsrätt, tvistelösning | Vid behov → offert per uppdrag |
| Energikonsult | Energideklaration, optimering | Lagkrav → upphandla specialistfirma |

**Karaktäristik:**
- Betalning: faktura, ibland abonnemang
- Utvärdering: kompetens, erfarenhet av BRF:er, personkemi, pris
- Kontakt: möten, digital kommunikation
- Aktörer: ordförande/kassör ofta mer involverade än vid fysisk

### 3. IT och digital infrastruktur

Digitala tjänster föreningen behöver.

| Tjänst | Typiska alternativ | Betalmodell | Beslutsfattare |
|--------|-------------------|-------------|----------------|
| **Bredband (fastighet)** | Telia, Bahnhof, Ownit, IP-Only | Per lgh/mån, 3–5 års avtal | Styrelsen (påverkar alla boende) |
| **E-post (styrelsen)** | Google Workspace, Microsoft 365, eget | Kort/faktura, per användare/mån | Styrelsen/admin |
| **Domän + hemsida** | Loopia, One.com, WordPress | Kort, årsavgift | Admin/ordförande |
| **Bokföringssystem** | Fortnox, Visma, via förvaltare | Abonnemang, faktura | Kassör + styrelsen |
| **Porttelefonapp** | Apt.us, Yale, 2N | Abonnemang + installation | Förvaltare + styrelsen |
| **Tvättstugebokning** | Electrolux, Aptus, eget | Leasing eller köp | Förvaltare |
| **Laddstolpar (drift)** | Charge Amps, Easee, Zaptec | Per kwh eller fast/mån | Styrelsen/stämman |
| **Digitalt styrelseverktyg** | Hemmet, Styrelseportalen, Boards | Abonnemang | Ordförande/styrelsen |

**Karaktäristik:**
- Betalning: ofta kort eller autogiro — **inte** bankgiro/faktura
- Utvärdering: funktionalitet, integration, support, skalbarhet, GDPR
- Kontakt: digital, onboarding, support
- Aktörer: den mest digitalt kunniga i styrelsen driver ofta, kassören hanterar betalning
- **PUB-avtal:** nästan alltid krav (persondata passerar tjänsten)
- **Bindningstid:** kan vara lång och dyr att bryta (bredband 5 år)

### 4. Finansiella tjänster

| Tjänst | Typiska alternativ | Beslutsfattare |
|--------|-------------------|----------------|
| **Fastighetslån** | Storbanker, SBAB, Landshypotek | Kassör utvärderar → styrelse/stämma beslutar |
| **Bankkonton** | Storbanker | Kassör + ordförande |
| **Swish företag** | Via banken | Kassör |
| **Företagskort** | Via banken, Pleo, Wallester | Kassör + ordförande |
| **Fakturahantering** | Bankgirot, Billogram, Fortnox | Kassör |

**Karaktäristik:**
- Betalning: avgifter, räntor
- Utvärdering: ränta, villkor, flexibilitet, digital service
- Kontakt: bankrådgivare, digital
- Aktörer: kassören driver, styrelsen/stämman beslutar
- **Bankbyte är komplext:** alla autogiro, bankgiro-nummer, firmateckning, fullmakter måste uppdateras

### 5. Försäkringar

| Försäkring | Alternativ | Beslutsfattare |
|------------|-----------|----------------|
| **Fastighetsförsäkring** | Länsförsäkringar, Trygg-Hansa, If | Ordförande/kassör → styrelsen |
| **Styrelseförsäkring** | Specialförsäkring, via förvaltare | Styrelsen |
| **Bostadsrättstillägg** | Via fastighetsförsäkring | Styrelsen |
| **Olycksfallsförsäkring** (frivillig) | Diverse | Stämman |

**Karaktäristik:**
- Betalning: årsvis, faktura eller autogiro
- Utvärdering: skydd, villkor, självrisk, pris, skadehanteringsprocess
- Kontakt: försäkringsrådgivare, mäklare
- Aktörer: kassör/ordförande utvärderar, styrelsen beslutar
- **Säsongsbetonat:** oftast förnyelse kring årsskiftet

### 6. Kommunal infrastruktur

Ofta begränsade valmöjligheter — leverantören bestäms av geografin.

| Tjänst | Valmöjlighet | Process |
|--------|-------------|---------|
| **El** | Elhandlare fritt val, elnät bestämt | Upphandla elhandelsavtal (fast/rörligt/mix) |
| **Fjärrvärme** | Normalt monopol (Stockholm Exergi etc.) | Förhandla villkor, inte välja leverantör |
| **Vatten/avlopp** | Kommunalt monopol | Ingen upphandling — taxa |
| **Sophantering** | Kommunalt eller upphandlat | Ibland valfrihet (ex. PMC, Ragn-Sells) |

### Aktörsrollerna varierar med kategori

| Kategori | Vem driver | Vem utvärderar | Vem beslutar |
|----------|-----------|----------------|-------------|
| Fysisk förvaltning | Förvaltare | Förvaltare (+konsult) | Styrelse/stämma |
| Tjänsteförvaltning | Ordförande/förvaltare | Ordförande + kassör | Styrelse |
| IT/digital | Mest IT-kunnig i styrelsen | Hela styrelsen (påverkar alla) | Styrelse |
| Finansiellt | Kassör | Kassör + ordförande | Styrelse/stämma |
| Försäkring | Kassör/ordförande | Kassör | Styrelse |
| Kommunalt | Förvaltare (förhandling) | — (monopol) | — (taxa) |

**Nyckelinsikt:** Det finns ingen "upphandlingsansvarig" i en BRF. Rollen delas mellan förvaltare (fysiskt), kassör (ekonomi/finans) och ordförande (strategiskt). Systemet behöver stödja alla tre som processägare.

## Livscykel — 10 faser

### Fas 0: Behov identifieras

**Triggers:**
- Besiktning med anmärkning → "vi behöver åtgärda X"
- Avtal som löper ut → "vi behöver nytt avtal för Y"
- Felanmälan som kräver entreprenör → "vi behöver VVS-firma"
- Stämmobeslut → "stämman har beslutat att vi ska Z"
- Styrelsediskussion → "vi borde utvärdera vårt bredband"
- Lagkrav → "OVK-besiktning krävs inom 6 mån"
- Underhållsplan → "fönsterbyte planerat 2027"

**Systemets roll:** Påminnelser från avtalsbevaking, besiktningsschema, underhållsplan.

### Fas 1: Specificering

Vad behöver vi? Kravformulering.

**Varierar per kategori:**

| Kategori | Specifikation | Omfattning |
|----------|--------------|-----------|
| Fysisk | Teknisk beskrivning, ritningar, besiktningsrapport | Detaljerad |
| Tjänste | Kravlista: vad ska ingå, SLA, rapportering | Medium |
| IT/digital | Funktionskrav, integration, GDPR, support | Medium |
| Finansiell | Lånebehov, amorteringsplan, räntebindning | Kassör/rådgivare |
| Försäkring | Skyddsnivå, självrisk, specialvillkor | Mäklare/direkt |

**Aktörer:** Den som driver kategorin + styrelsen vid större upphandlingar.

### Fas 2: Marknadsundersökning / leverantörsval

Vilka finns? Vilka ska vi fråga?

**Källor:**
- Leverantörsregistret i Hemmet
- Rekommendationer från HSB/Riksbyggen/andra BRF:er
- Branschorganisationer
- Tidigare erfarenhet
- Digitala marknadsplatser (för IT-tjänster)
- Bankrådgivare (för finansiella)
- Försäkringsmäklare (för försäkringar)

**Antal att tillfråga (tumregel):**
- < 50 000 kr: 1–2 st
- 50 000–500 000 kr: 2–3 st
- > 500 000 kr: 3–5 st
- Storskaligt: formell upphandling

**Jävskontroll:** Innan förfrågan skickas — har någon styrelsemedlem koppling till tillfrågad leverantör?

### Fas 3: Offertförfrågan (RFQ)

Formell eller informell förfrågan.

| Kategori | Förfråganformat |
|----------|----------------|
| Fysisk | E-post med kravspec + besiktningsrapport, platsbesök |
| Tjänste | Möte + skriftlig förfrågan |
| IT/digital | Testperiod, demo, funktionsjämförelse |
| Finansiell | Möte med bankrådgivare, skriftlig offertbegäran |
| Försäkring | Offertförfrågan via mäklare eller direkt |

### Fas 4: Offertinsamling

Offerterna inkommer. Systemet stödjer:
- Auto-koppling via e-post (plus-adressering)
- Påminnelse vid deadline
- Registrering av offertdata (belopp, villkor, tidsplan)
- Bilagor i dokumentarkivet

### Fas 5: Jämförelse och utvärdering

**Utvärderingskriterier varierar:**

| Kriterium | Fysisk | Tjänste | IT/digital | Finansiell | Försäkring |
|-----------|--------|---------|-----------|------------|------------|
| Pris | ●● | ●● | ●● | ●●● | ●● |
| Kvalitet/kompetens | ●●● | ●●● | ●● | ● | ● |
| Garanti | ●●● | ● | — | — | — |
| Referensobjekt | ●●● | ●● | ● | — | — |
| Tidsplan | ●●● | ● | ● | ● | — |
| Support/SLA | ● | ●● | ●●● | ● | ●● |
| Integration | — | ● | ●●● | ●● | — |
| GDPR/PUB | ● | ●● | ●●● | ●● | ● |
| Bindningstid | ● | ●● | ●●● | ●●● | ● |
| Uppsägningstid | ● | ●● | ●● | ●● | ● |
| F-skatt | ●●● | ●● | ● | — | — |
| Försäkring | ●●● | ● | — | — | — |
| Betalmetod | ● | ● | ●● | — | ● |
| Skadehantering | — | — | — | — | ●●● |

### Fas 6: Beslut

| Belopp/typ | Beslutsnivå |
|-----------|-------------|
| Akut reparation < delegationsgräns | Förvaltare |
| Driftsavtal inom budget | Styrelsen |
| IT-tjänst (påverkar alla) | Styrelsen |
| Lån/omfinansiering | Stämman |
| Storskaligt underhåll (> budget) | Stämman |
| Nyinvestering (laddstolpar etc.) | Stämman |

**Systemets roll:** Koppla beslut till upphandling. Jävsdeklaration. Protokollföring.

### Fas 7: Avtalstecknande

Upphandlingen resulterar i ett avtal (→ Contract-modellen).

**Betalmetod bestäms här:**
- Bankgiro/faktura (fysisk, tjänste)
- Autogiro (el, försäkring, bank)
- Kort (IT/SaaS, domän, småtjänster)
- Direkt via förvaltare (Fortnox, via banken)

### Fas 8: Utförande / onboarding

| Kategori | "Utförande" betyder |
|----------|-------------------|
| Fysisk | Arbete på fastigheten — inspektion, godkännande |
| Tjänste | Onboarding, överlämning, tillgångar |
| IT/digital | Setup, konfiguration, migration, utbildning |
| Finansiell | Kontoskapande, fullmakter, autogiro |
| Försäkring | Aktivering, registrering av fastighetsdata |

### Fas 9: Godkännande

| Kategori | Godkännande av |
|----------|---------------|
| Fysisk | Förvaltare inspekterar + slutbesiktning |
| Tjänste | Löpande kvalitetsgranskning, SLA-uppföljning |
| IT/digital | Funktionstester, driftsättning |
| Finansiell | Bekräftelse av villkor, första transaktion |
| Försäkring | Bekräftelse av skyddsnivå |

### Fas 10: Löpande uppföljning + garanti

- Garantibevakning (fysisk)
- SLA-uppföljning (tjänste, IT)
- Kostnadsuppföljning (alla)
- Avtalsbevakning (alla)
- Nästa upphandlingscykel

## Datamodell

Upphandlingsmodellen behöver vara generisk nog att hantera alla kategorier:

```prisma
enum ProcurementCategory {
  PHYSICAL          // Fysisk förvaltning (VVS, el, målare, tak)
  SERVICE           // Professionella tjänster (förvaltning, revision, juridik)
  IT_DIGITAL        // IT och digital infrastruktur
  FINANCIAL         // Finansiella tjänster (bank, lån)
  INSURANCE         // Försäkringar
  UTILITY           // Kommunal infrastruktur (el, värme, sopor)
}

enum ProcurementStatus {
  DRAFT                // Under planering
  SPECIFICATION        // Kravspec under utformning
  RFQ_SENT             // Offertförfrågan skickad
  COLLECTING_QUOTES    // Offerter inkommer
  EVALUATION           // Utvärdering / jämförelse
  DECISION_PENDING     // Inväntar beslut (styrelse/stämma)
  DECIDED              // Beslut fattat
  CONTRACTING          // Avtal under tecknande
  ONBOARDING           // Leverantör onboardas / arbete påbörjat
  COMPLETED            // Slutförd, avtal tecknat
  CANCELLED            // Avbruten
}

model Procurement {
  id                String              @id @default(cuid())
  title             String
  description       String?             @db.Text
  category          ProcurementCategory
  status            ProcurementStatus   @default(DRAFT)

  // Budget
  estimatedCost     Float?              // Uppskattad kostnad
  actualCost        Float?              // Faktisk kostnad (vid avslut)
  budgetApproved    Boolean             @default(false)
  isRecurringCost   Boolean             @default(false) // Sant för driftsavtal
  annualEstimate    Float?              // Estimerad årskostnad (om löpande)

  // Tidplan
  rfqSentAt         DateTime?
  quotesDeadline    DateTime?
  plannedStart      DateTime?           // Planerat startdatum för tjänst/arbete
  plannedEnd        DateTime?           // Projektslut (null = löpande)
  actualStart       DateTime?
  actualEnd         DateTime?

  // Beslut
  mandateLevel      String?             // DELEGATED, BOARD, ANNUAL_MEETING
  decisionId        String?             // Styrelsebeslut
  decisionNote      String?  @db.Text   // Motivering (viktigt vid revision)

  // Resultat
  selectedQuoteId   String?
  contractId        String?             // Avtal som skapades av upphandlingen

  // Garanti (för fysiska)
  warrantyMonths    Int?
  warrantyExpiry    DateTime?

  // Kopplingar — vad triggade upphandlingen
  triggerType       String?             // "Inspection", "DamageReport", "Contract", "Decision"
  triggerId         String?
  triggerTitle      String?             // Läsbar: "Fasadbesiktning 2025", "Städavtal löper ut"

  // Process
  createdById       String
  ownerId           String              // Vem driver upphandlingen (förvaltare/kassör/ordförande)

  createdAt         DateTime            @default(now())
  updatedAt         DateTime            @updatedAt

  createdBy         User                @relation("ProcurementCreator", fields: [createdById], references: [id])
  owner             User                @relation("ProcurementOwner", fields: [ownerId], references: [id])
  quotes            ProcurementQuote[]
  evaluationNotes   ProcurementNote[]

  @@index([status])
  @@index([category])
  @@index([ownerId])
}

model ProcurementQuote {
  id              String   @id @default(cuid())
  procurementId   String
  contractorId    String?

  // Leverantörsinfo (om ej i Contractor)
  companyName     String
  orgNumber       String?
  contactPerson   String?
  contactEmail    String?
  contactPhone    String?

  // Offert
  amount          Float?              // Engångsbelopp eller total
  amountExVat     Float?
  annualCost      Float?              // Årskostnad (för löpande)
  monthlyCost     Float?              // Månadskostnad (IT/SaaS)
  currency        String   @default("SEK")
  validUntil      DateTime?

  // Villkor
  proposedStart   DateTime?
  proposedEnd     DateTime?
  contractLength  Int?                // Avtalslängd i månader
  noticePeriod    Int?                // Uppsägningstid i månader
  warrantyMonths  Int?
  paymentTerms    String?             // "30 dagar netto", "Kort", "Autogiro"
  paymentMethod   String?             // INVOICE, CARD, AUTOGIRO, PREPAID
  conditions      String?  @db.Text
  
  // Utvärdering
  status          String   @default("PENDING") // PENDING, RECEIVED, SELECTED, REJECTED
  receivedAt      DateTime?
  
  // Bilagor via Attachment-modellen (entityType: "ProcurementQuote")

  createdAt       DateTime @default(now())

  procurement     Procurement @relation(fields: [procurementId], references: [id], onDelete: Cascade)
  contractor      Contractor? @relation(fields: [contractorId], references: [id])

  @@index([procurementId])
}

// Interna noteringar under utvärdering
model ProcurementNote {
  id              String   @id @default(cuid())
  procurementId   String
  authorId        String
  content         String   @db.Text   // "Kassören: ryms inte i budget utan reservfond"
  createdAt       DateTime @default(now())

  procurement     Procurement @relation(fields: [procurementId], references: [id], onDelete: Cascade)
  author          User        @relation(fields: [authorId], references: [id])

  @@index([procurementId])
}
```

## Jävskontroll

Upphandlingar är den vanligaste jävssituationen i en BRF.

### Kontrollpunkter

| Fas | Kontroll |
|-----|----------|
| Fas 2 (leverantörsval) | Har någon styrelsemedlem koppling till tillfrågad leverantör? |
| Fas 5 (utvärdering) | Jävig ledamot deltar inte i utvärdering |
| Fas 6 (beslut) | Jävig ledamot lämnar rummet vid omröstning |
| Fas 7 (avtal) | Jävig ledamot signerar inte avtalet |

### Systemets roll

```
⚠ Jävskontroll — UPH-2026-003 Ommålning fasad

  Offerande leverantörer:
  ✅ Målare Andersson AB — ingen styrelsemedlem har registrerad koppling
  ⚠ Fasadspecialisten AB — Erik Larsson (förvaltare):
     "Bengt Fasad är min svåger" (registrerad 2025-08-01)
     
  Erik bör inte delta i utvärdering eller beslut.
  
  [Registrera jävsdeklaration]
```

  - Fasadmålning: offertjämförelse pågår, 3 offerter
